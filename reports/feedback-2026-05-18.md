# 使用回饋改善草案 — 2026-05-18

來源:Notion「Workflow Improvement Suggestions」
(https://www.notion.so/3648ebada945815b8d94e90f876d8c5b),建立於 2026-05-18,
觸發 session 為 SRC Batch sepasoft-foundations 校準輪。

範圍:本草案只處理與 `sepasoft-foundations` skill 內容直接相關的部分
(回饋 §3)。已忽略非本 repo skill 的工作流問題:§1.2 notion-mcp skill
載入、§4.1 compact 摘要 template、§4.2 system prompt 強制力 —— 這些屬
harness / 平台層,不在兩個 foundations skill 的維護範圍。

狀態:草案,待 user review 後逐項放行才動 skill 檔案。本檔不修改任何
skill 交付物。

## 核心結論

回饋 §3 把多個主題列為「強烈建議新開 reference」。與現有 references 逐項
比對後,**多數內容 skill 已具備**。回饋 §3.4 自己預警了這個重複風險,實
際情況印證:真正的 gap 比回饋估計小很多,且不需要新開任何檔。

設計定位約束(見 CLAUDE.md):這兩個 skill 是 conceptual grounding,不是
troubleshooting / API 速查庫。新增內容前須通過來源紀律 self-check(有合格
官方來源、來源實際支持陳述、版本特定則標版本);無來源則標待補,不憑記憶
或一份使用回饋補斷言。

優先度:中。

### B. 回饋建議新開 `batch-state-machine.md`(§3.1)— 否決新開,僅補誤區三點

### C. 回饋建議改寫 `ebr-data-model.md`(§3.1)— 部分採納,first/last 升級為 sourced

回饋聲稱「提取規則:預設 last occurrence、BEGIN_DATE_TIME 例外用 first」
有「3.81.8 Release Note 背書」,並要求補 8 個 main parameter 清單、三類
寫入機制、SP6+ `{first(X)}`/`{last(X)}` template syntax。

來源驗證分兩階段。第一階段(只查 docs user-manual 頁)未找到 verbatim,
草案初版據此判「維持 unverified」。**該判定是錯的** —— 它把搜尋範圍限在
user-manual 頁,沒查 Release Notes Publication。user 隨後提供確切 URL
後第二階段查證推翻了它。據實記錄如下,不靜默改立場。

第二階段(2026-05-18,Playwright 實查 user 提供的
https://docs.sepasoft.com/articles/release-notes-publication/3-81-8-release-notes/):

- **3.81.8 SP6**(verbatim):「EBR Viewer component no longer displays
  referenced parameters as "null" or the incorrect occurrence. **All
  parameters except BEGIN_DATE_TIME use the last occurrence of the
  parameter**」—— 直接支持「預設 last、BEGIN_DATE_TIME 例外用 first」。
- **3.81.8 SP6**(verbatim, ADDED):「EBR Viewer template string can use
  `{first(PARAM_NAME)}` or `{last(PARAM_NAME)}` to specify if the first
  or last occurrence of a parameter value should be used in the detail
  section」—— 支持 SP6 template syntax,版本切點為 3.81.8 SP6。
- **3.81.8 base**(verbatim, ADDED):「`system.mes.batch.queue.getExecutedBatchIDs`
  script method for getting a list of Batch IDs for batches that have
  already been executed and removed from the Batch Queue」—— 確認該方法
  自 3.81.8 引入,且其語意佐證 Active queue / Executed(已從 queue 移除)
  兩階段互斥 lifecycle(關聯 A 項)。

裁決:已取得 docs.sepasoft.com 合格官方來源,**C 的 first/last 規則與
SP6 template syntax 從 unverified 升級為 sourced**。改寫 ebr-data-model.md
時須精確化,不可放寬措辭:

- first/last 規則的 verbatim 範圍是 **EBR Viewer component / template
  string 行為**,版本標 **3.81.8 SP6**;不可泛化成「任何 EBR 讀取路徑都
  套此規則」。`getParameterValue` 的 live-vs-persisted 規則是另一回事,
  維持原記載。
- `{first()}`/`{last()}` template syntax 標 3.81.8 SP6 起,來源 URL 為
  上述 Release Notes。
- 仍**無**官方 verbatim 的部分:回饋要求的「8 個 main parameter 清單」
  與「三類寫入機制」此 Release Notes 頁未列出,user-manual 頁也無。這
  兩項維持待補,不因 first/last 已驗證就連帶寫入。

優先度:中(來源已具備;改寫須嚴格綁定 EBR Viewer 範圍與 SP6 版本)。

## 修正記錄(epistemic)

草案初版對 C 判「維持 unverified、不動」是錯的。錯因:來源查證只掃了
docs user-manual 區,未涵蓋 Release Notes Publication 區,就下了「官方
無 verbatim」的結論。user 提供確切 Release Notes URL 後複查,3.81.8 SP6
有直接 verbatim 支持。正確結論是 C 升級為 sourced(範圍限定 EBR Viewer
+ SP6)。教訓:查證「官方有無此陳述」時,Release Notes Publication 與
user-manual 是兩個獨立區,版本特定行為(SP-gated)優先查 Release Notes,
不能只掃 user-manual 就斷言「官方未記載」。
